Root Test Pages

These pages primarily test standard Autofill scenarios within a single frame or with simple embedding.

  • address-forms.html Partial Address Forms: Contains multiple independent "micro-forms" (e.g., Name only, Address only, City only) to test how Autofill handles incomplete address sections.
  • address_form_with_defaultprevented.html JS-Intercepted Submission: An address form where the submit event is intercepted and preventDefault() is called, testing if Autofill correctly detects submission when the browser's default behavior is blocked.
  • big-address-form.html Comprehensive Address: A standard, large layout containing all typical address fields (Name, Lines 1-3, City, State, Zip, Country) to verify full-profile filling.
  • email-login-form.html Standard Login: A basic email + password login form.
  • full-address-form.html Configurable Submission: A versatile address form that uses URL parameters (preventDefault, redirectWhenDefaultPrevented) to dynamically toggle test behaviors for submission and redirection.
  • password-eval-html-submission-effect.html Dynamic Form Lifecycle: Creates a password form via JavaScript and removes it immediately upon submission preventDefault. Tests if Autofill can capture the "submission" signal even if the form is destroyed DOM-synchronously.
  • password.html Submit by Deletion: A password form that includes a button to trigger submission by removing the form element from the DOM (a pattern seen in some Single Page Apps).
  • post-message.html Msg Communication: A utility page that accepts and replies to postMessage events. Useful for verifying cross-origin communication channels.
  • signup-form.html Registration: A signup flow with "Username", "New Password", and "Confirm Password" fields.
  • singe-cc-field-form.html Partial Credit Card: A form containing *only* a Credit Card Number field, testing heuristics for single-field forms.
  • single-email-field-login.html Step-based Login: A login flow with only an Email field (e.g., "Enter email first" flows).
  • submit-only-form.html Empty Form: A form with only a submit button, testing robust handling of empty contexts.
  • svg-anchor.html Extraction Edge Case: Tests a specific bug where extraction scripts might fail on SVG <a> tags because their href property is an object (SVGAnimatedString), not a string.
  • synthetic-form.html Autocomplete Attributes: A simple form ensuring that fields are identified correctly strictly via their autocomplete attributes.

Autologin Test Pages

Pages specifically designed to test Password Manager's auto-submission heuristics.

  • autologin/auto_submit_test.html Standard Auto-Submit: This form meets the criteria for Password Manager auto-submission.
  • autologin/autologin-form.html Legacy Auto-Submit: A login form designed to satisfy Password Manager auto-submission heuristics.
  • autologin/auto_submit_with_preceding_field.html Preceding Field: This form SHOULD auto-submit. The heuristic allows focusable fields BEFORE the username.
  • autologin/auto_submit_disabled_field_after.html Disabled - Field After: This form should NOT auto-submit because there is a focusable text field AFTER the password.
  • autologin/auto_submit_disabled_field_between.html Disabled - Field Between: This form should NOT auto-submit because there is a focusable text field between the username and password.
  • autologin/auto_submit_disabled_radio_button.html Disabled - Radio: This form should NOT auto-submit because Radio Buttons are NOT ignored by the heuristic, unlike Checkboxes.

XFrame Test Pages

These pages are critical for testing Autofill Across Iframes (iOS feature), where a single logical form is split across multiple documents (iframes).

  • xframe/address-and-cc.html Hybrid Forms: Tests a page containing both an Address form and a Credit Card form hosted in different iframes.
  • xframe/address-form-xframe.html Distributed Address: An address form where individual fields (Name, Address Line) reside in separate iframes.
  • xframe/cc-form.html Standalone / Child: A full credit card form. Often used as a child frame source, but runnable independently.
  • xframe/formless_index.html Formless Aggregation: A set of iframes containing credit card fields (Name, Number, Exp, CVC) *without* a wrapping <form> tag in the parent document. Tests heuristic aggregation.
  • xframe/formless_index2.html Formless Variation: Similar to the above, but testing a different layout/grouping of the unowned fields.
  • xframe/index-srcdoc.html Srcdoc Support: Tests cross-frame autofill when one of the child frames is created via data in srcdoc (which has a unique origin handling).
  • xframe/index.html Standard Split-Form: The primary test case. A parent form contains the Name field, while logic requires bridging to child iframes for Number, Expiry, and CVC.
  • xframe/index2.html Mixed Layout: A variation of the split-form where the Name field is removed from the parent, leaving mostly iframe containers and a <select> for Expiry in the parent.
  • xframe/target-test.html Target Attribute: Tests a form submitted with target="target-frame", verifying that Autofill handles submission signals correctly when the response is directed to a sibling iframe.
  • xframe/xhr-problem-repro-multiframes.html Multi-Frame XHR: Reproduces a specific bug where an AJAX submission removes forms from multiple child iframes, testing if the "submission detected" signal survives the destruction.
  • xframe/xhr-problem-repro.html Single-Frame XHR: A simpler variant of the XHR destruction test case with a single child iframe.
